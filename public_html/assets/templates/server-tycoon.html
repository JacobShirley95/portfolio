What it is

Server Tycoon is a game I helped to make as part of a group project at university. We had to design a game for KS2 or KS3 students that would aid there learning in some part of the curriculum. We chose computing to be the part of the curriculum that our game would target.

The game follows a similar structure to other tycoon games, and has the main aim of trying to host as many clients as possible on a server to make as much money as possible. You also have to complete small tasks for the client to keep them happy and increase your reputation, whereby you can then take on more high-paying clients. These will require better servers, however.

Features

  • Choice of servers which can be placed anywhere in the world
  • Servers can be configured, overclocked and upgraded
  • Economy and time
  • Client generation
  • Server fires
  • Some scenarios for the clients (more to be added)

My Contribution

My main tasks on the project were to get the server and client systems implemented.

I started off with server placement, giving them a red overlay when they weren’t placeable, and turning them back to normal when they were. This worked out nicely, and each server can be placed anywhere in the room, as long as it’s not on top of anything and allows enough space for the door.

I then moved onto the server configuration screen. The interface for this was designed by me using the relatively limited Unity user interface tools – “limited” is the best way to describe them coming from an HTML/CSS background. All the buttons and sliders are functional, and the upgrades actually affect the performance of the server.

This is the main configuration screen I designed for the servers. It is the main hub where players can check the stats of their server, overclock it, and upgrade it.

Clients hosted on this server can be selected from a list and have their current server usage displayed, and then optionally removed (-10 reputation per client removed).

Ports can be opened, which are required by clients, at the expense of lowered server security (-5% per port opened).

The server can be renamed or sold from the panel at the top.

The next part was to get the clients generated and loaded onto the server. Originally clients were going to be predefined in a JSON file, but we quickly realised this would negatively impact replayability. Opting for the randomly generated option, I made a word generator which takes a random adjective and noun and puts them together to create a name for a server or client. This can often have hilarious results.

This is the client selector interface. Server attributes that the client requires are randomly generated and displayed in each field (type, processor, etc).

From easy to hard the values get more difficult and require more server power. If the player's reputation is too low some types of client cannot be taken on.

Clients can also be declined, but the player will have to wait around 10 seconds for a new one to be generated.

Once the client generation was done, I had to make a user interface to display them in. I designed this user interface myself so it can automatically extend to any number of client-difficulty types added. Other parts of the project I designed are the server purchase list, the client list in the bottom right of the screen, and the Python programming scenario, which uses an actual Python interpreter for C# known as “Iron Python”.

The last large part of the game I worked on was the tutorial – the guy called Pan who follows the player through the systems of the game, explaining what each interface does, and some of the concepts behind the curriculum (though the main information about each subject is in the bookshelf). Throughout the whole project, I was also constantly fixing bugs and adding small improvements here and there.

Teamwork

From the start of this process we did not really have a true leader for the project, which worked relatively well most of the time as we could all do whatever job we wanted. This is similar to how the Agile methodology works, where there is a lot of fluidity in how tasks are assigned to programmers. We would usually just decide an outline of the work we’d each individually do. For instance, I said I would do the server placements, and that grew into the whole of the server functionality and clients as well. Some parts of the project, however, could have benefited from some more predefined strategies from the start, such as project structuring so we would know how to organise our code and not accidently overwrite some of the features that someone else on the project had added.

Reflection

As said above, I think a leader would have been helpful during parts of the project. I learned how to use Unity for the first time. Despite the ups and downs throughout, I am very proud of the work we have achieved in a limited time span, without any pre-existing knowledge of the Unity environment, and never having worked on a project of this scale.
